# Folder Structure

KBForge uses a numbered folder convention. The numbers create a natural workflow order: raw input → structured knowledge → working output → logs.

---

## Overview

```
kbforge-starter/
├── AGENTS.md                    → Root agent runbook
├── 00 Raw Sources/              → Unprocessed inputs
│   ├── AGENTS.md
│   └── Archive/                 → Processed sources (audit trail)
├── 01 Reference/                → Structured KB articles
│   ├── AGENTS.md
│   ├── articles/                → One .md per article
│   ├── aggregates/              → Auto-generated cross-cutting views
│   ├── catalog.jsonl            → Machine-readable article index
│   ├── catalog.md               → Human-readable table of contents
│   ├── topics.yaml              → Controlled vocabulary for domains/topics
│   └── scripts/                 → Python build scripts
├── 02 Outputs/                  → Working drafts, issue notes, replies
│   └── AGENTS.md
├── 03 Accounts/                 → Lightweight partner/client context cards
│   └── AGENTS.md
├── 04 Skills/                   → Reusable AI workflow definitions
│   ├── AGENTS.md
│   ├── skills/                  → Individual skill .md files
│   ├── scripts/                 → build_skills_index.py
│   └── aggregates/              → Skill cross-references
├── 05 Iteration Logs/           → Append-only logs
│   ├── AGENTS.md
│   ├── KB Gaps.md               → Unanswered questions and coverage gaps
│   └── Ingestion Log.md         → Record of every ingestion
```

---

## Folder details

### `00 Raw Sources/`

Drop any unstructured text here: transcripts, API docs, meeting notes, chat logs, emails. The `clean-source` and `ingest-source` skills process these into structured articles.

After processing, sources are moved to `Archive/` with updated frontmatter (`processed: true`). This creates an audit trail — you can always trace an article back to its origin.

### `01 Reference/`

The core of your knowledge base. Every article is a Markdown file with structured frontmatter. The `articles/` folder holds one file per article. The `aggregates/` folder holds auto-generated views like glossaries and cross-references.

`catalog.jsonl` is the machine index. `catalog.md` is the human index. Both are regenerated by `build_catalog.py`.

`topics.yaml` defines your controlled vocabulary. Every `domain:` and `topics:` value in article frontmatter must exist here. This prevents tag sprawl and keeps search predictable.

### `02 Outputs/`

Working space for drafts, issue notes, reply templates, and anything ephemeral. Files here are not part of the permanent KB. Clean up regularly.

### `03 Accounts/`

Lightweight context cards for partners, clients, or stakeholders. One file per account. The `draft-response` skill reads these before writing partner-specific replies.

### `04 Skills/`

Reusable AI workflows. Each skill is a Markdown file with frontmatter (triggers, inputs, outputs) and a procedure section. The AI loads these dynamically based on your request.

`skills.jsonl` is the skill index, analogous to `catalog.jsonl`. `build_skills_index.py` regenerates it.

### `05 Iteration Logs/`

Append-only logs. Never rewrite history.

- `KB Gaps.md` — Questions the AI couldn't answer, coverage gaps, open issues
- `Ingestion Log.md` — Record of every source processed

These logs drive continuous improvement. Review `KB Gaps.md` regularly to find what's missing from your KB.

## The `AGENTS.md` pattern

Every folder has an `AGENTS.md` file. This is the local runbook — it tells the AI:

- What this folder is for
- What rules apply here
- Which skills to use
- What not to do

Before working in any folder, the AI reads its `AGENTS.md`. This keeps behavior consistent and scoped.
